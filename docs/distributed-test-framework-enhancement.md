# 分布式测试框架增强计划

本文档面向后续测试基础设施的研发实现者，用于把 roadmap 中“分布式测试框架增强”从方向性目标细化为可执行设计。它不替代 [复制语义专题文档](/root/dev/sys/turntf/docs/replication-semantics.md) 和 [分布式系统未来演进路线图](/root/dev/sys/turntf/docs/distributed-system-roadmap.md)，而是回答“现有测试基础是什么、下一步要抽什么夹具、按什么顺序落地、什么算完成”。

## 背景与目标

当前仓库已经具备以下分布式测试基础：

- `internal/store` 已覆盖字段级 `LWW`、删除墓碑、消息窗口、订阅复制、幂等吸收等收敛语义。
- `internal/cluster/manager_test.go` 已覆盖握手、`Ack`、补拉、快照修复、校时保护、动态路由、多跳查询等协议路径。
- 测试中已经存在真实多节点场景：通过 `net.Listener + HTTP API + WebSocket + Store + Manager` 启动两个或三个节点，验证复制收敛和路由行为。

这些基础已经足以支撑当前实现，但仍存在明显短板：

- 多节点测试夹具散落在 `internal/cluster/manager_test.go` 末尾，复用方式偏“复制已有 helper 再局部改造”。
- 多数复杂场景依赖用例内直接拼装监听器、节点启动、轮询等待和 HTTP 请求，导致新增场景成本偏高。
- 当前夹具更擅长“真实链路 happy path + 少量故障路径”，对网络分区、乱序、重复投递、丢包、并发快照等故障注入支持不足。
- 测试覆盖已经触达复制语义，但尚未形成清晰的“三层测试策略 + 固定回归集 + 新能力接入规则”。

本计划的目标不是一次性重写全部测试，而是在不改变现有语义承诺的前提下，把已有测试资产提炼成统一的分层框架，让后续黑名单、mTLS、自动发现、时钟状态机等能力可以“先补夹具、再落实现、最后纳入回归”。

## 设计原则

- 先抽象测试职责，再补新场景；不要把业务能力改动和测试框架重写绑在同一轮。
- 统一夹具必须显式对齐 [复制语义专题文档](/root/dev/sys/turntf/docs/replication-semantics.md) 的术语，断言优先表达“收敛语义”而不是局部实现细节。
- 多节点测试框架应按“消息收发 + 连接生命周期 + 故障注入点”建模，不把 WebSocket 细节硬编码为唯一传输假设。
- 优先收编现有 helper 与回归用例，避免为了框架整洁而重写一整套不同风格的测试模型。
- 新增基础设施只服务仓库内部测试，不新增对外公共 API，也不引入外部测试编排系统。

## 当前测试版图

### 1. `store` 层收敛测试

这一层已经具备较好的语义聚焦，主要验证：

- 用户、订阅和消息投影在重复事件、乱序应用下仍可收敛。
- 删除墓碑阻止旧事件复活。
- `message_window_size` 对本地窗口裁剪和最终结果的影响。
- 事件日志、投影和幂等表之间的一致性。

这一层后续仍应保持“不引入网络细节”的边界，只关注状态演化与收敛规则。

### 2. `cluster manager` 协议测试

当前这一层主要集中在 [internal/cluster/manager_test.go](/root/dev/sys/turntf/internal/cluster/manager_test.go)，已覆盖：

- 握手合法性、版本兼容与方向裁决。
- `Ack` 语义、补拉请求构造、快照摘要与分片请求。
- 未来时间戳拒绝、首次校时前写闸门、最大时钟偏差保护。
- 动态路由、多跳瞬时包和登录用户查询。

这一层的问题不是“没有覆盖”，而是缺少统一抽象。当前既有纯协议级测试，也混入了真实网络测试和 API 驱动测试，职责边界不够清晰。

### 3. 多节点仿真测试

当前仓库已经拥有多节点集成测试雏形，例如：

- 双节点复制与 `Ack`
- 双节点相同/不同窗口大小下的消息收敛
- 晚加入节点补拉追平
- 快照修复收敛
- 三节点字段级收敛与删除优先
- 多跳瞬时包路由与登录用户查询

这类测试已经证明“真实节点启动 + 真实 HTTP/WebSocket”路径是可行的，但目前缺少统一的场景驱动器，导致以下问题：

- 节点创建、启动、轮询等待、日志采集散落在 helper 和用例里。
- 难以系统性注入网络故障，更多依赖自然重连而不是受控模拟。
- 新能力落地时，容易继续把测试写成“单个大用例 + 一次性私有 helper”。

## 目标测试架构

增强后的测试体系分为三层，每层都必须有明确职责和准入规则。

### 1. `store` 层：状态收敛测试

职责：

- 验证字段级 `LWW`、墓碑、消息窗口、订阅可见性、幂等吸收等本地状态规则。
- 验证快照分片或事件重放进入存储层后的合并结果。

不负责：

- WebSocket、握手、路由、重连、真实网络故障。

推荐实现：

- 保持当前基于 `store.Open` 和本地仓储的测试模式。
- 把后续黑名单、向量时钟冲突元数据等能力优先拆成可在本层独立断言的状态规则。

### 2. `cluster manager` 层：协议状态机测试

职责：

- 验证握手、`Ack`、补拉、快照、校时、路由、错误路径和状态机转换。
- 验证“某协议输入产生何种本地状态变化、回包和日志”。

不负责：

- 大规模多节点拓扑收敛，也不直接承诺端到端最终一致。

推荐实现：

- 继续保留现有 session 级 helper。
- 补出可控 transport/session stub，用于注入消息顺序、失败、超时、时间样本和快照推进。
- 让协议测试优先依赖 stub，而不是每个错误路径都启动真实 WebSocket。

### 3. 多节点仿真层：端到端分布式行为测试

职责：

- 启动真实节点，验证网络故障、重连、反熵、快照并发、membership 变化、路由传播下的端到端行为。
- 验证文档承诺的最终一致语义，而不是某条局部函数路径。

推荐实现：

- 在保留真实 HTTP / WebSocket / Store / Manager 组合的前提下，增加统一的场景驱动器和故障注入点。
- 把“网络行为控制”放在夹具层，而不是散落在具体业务测试里。

## 统一夹具设计

统一夹具默认拆为三类内部测试基础设施。

### 1. 节点测试句柄

职责：

- 封装单节点的构造、启动、关闭和基础观测。
- 提供本地 API 操作入口，例如 HTTP/JSON 请求、客户端 WebSocket 登录、管理端状态读取。
- 暴露必要的只读观测，例如 peer 活跃状态、最后 `Ack`、当前 session、日志缓冲区、节点基础地址。

建议能力：

- 可配置 `node_id`、`message_window_size`、时钟偏移、`max_clock_skew_ms`。
- 可显式控制“是否启动 manager / HTTP server / routing 能力”。
- 默认自带日志采集，便于断言 join、reconnect、catchup、snapshot 和时钟拒绝等事件。

### 2. 集群场景驱动器

职责：

- 负责多节点拓扑搭建、节点间连接关系和故障注入。
- 统一提供“建立集群、切断链路、恢复链路、延迟消息、重放消息、丢弃消息、触发重连、推进快照”的场景 API。

建议能力：

- 支持两节点、三节点和链式/星型拓扑的快速搭建。
- 支持网络分区、乱序、重复投递、丢包、延迟、重连等故障模型。
- 支持控制首次校时结果、后续校时偏移、补拉批次和快照分片推进节奏。

抽象边界：

- 驱动器以“消息收发和连接生命周期”为抽象边界，不以 WebSocket 帧细节为主抽象。
- 如果底层仍使用真实 WebSocket，驱动器也应把“断链/延迟/重放”包装为通用动作，避免未来传输抽象时重写全部场景测试。

### 3. 语义断言集

职责：

- 把当前测试里零散的轮询与比较逻辑收敛为高层断言。
- 让测试描述直接贴近复制语义，例如“最终收敛”“窗口保留 N 条”“删除优先于并发更新”“`Ack` 表示已应用到本地状态”。

建议能力：

- `EventuallyConvergesUsers`
- `EventuallyConvergesMessages`
- `PeerAckReached`
- `SnapshotRepairCompleted`
- `WriteGateBlockedUntilClockTrusted`
- `RouteBecomesReachable`

这些名字可以在实现时调整，但职责必须稳定：断言表达语义，不暴露底层轮询细节。

## 分阶段落地

### Phase 1：抽取现有夹具，不改语义

目标：

- 从 [internal/cluster/manager_test.go](/root/dev/sys/turntf/internal/cluster/manager_test.go) 提炼现有节点构造器、等待函数、HTTP helper、日志采集 helper。
- 形成统一测试支撑层，替换“每个复杂场景单独拼装”的模式。

交付物：

- 单节点测试句柄初版。
- 通用等待/断言工具初版。
- 对现有关键场景零语义改动迁移，确保只是搬运夹具，不改变测试预期。

完成标准：

- 新增多节点测试时，不再直接在业务用例中手写监听器、HTTP server 启动和基础轮询逻辑。

### Phase 2：收编关键回归场景

目标：

- 把现有已验证的核心语义收编为固定回归集，而不是继续分散在一次性测试里。

固定回归集至少包括：

- 双节点复制、补拉、重复投递幂等。
- 晚加入节点追平、跨多批次补拉、快照修复收敛。
- 相同与不同 `message_window_size` 下的窗口行为。
- 三节点并发写入的字段级收敛与删除优先。
- 多跳路由与登录用户查询。
- 时钟超限、未来时间戳拒绝、首次校时前写闸门。

完成标准：

- 新增协议特性时，设计评审可以明确指出它应补到哪一层、哪一组回归集，而不是新增孤立用例。

### Phase 3：补齐故障注入与多节点仿真

目标：

- 让框架具备主动制造复杂分布式场景的能力，而不是只验证自然路径。

新增覆盖至少包括：

- 网络分区与恢复。
- 乱序到达、重复投递与补拉重放。
- 实时广播与补拉并发。
- 快照请求、快照应用与事件流并发。
- membership 抖动、重复地址或链路抖动下的收敛与恢复。

完成标准：

- 多节点仿真测试可复用相同驱动器表达故障场景，而不是为每个场景再造一套私有控制逻辑。

### Phase 4：为后续能力预留扩展点

目标：

- 让黑名单、mTLS、自动发现等后续能力能够在现有测试框架中接入，而不是再次逼出新的测试风格。

必须预留的扩展入口：

- 身份与握手扩展：支持未来证书身份、双向认证、额外握手元数据断言。
- membership 变化：支持动态 peer 加入、移除、重复地址和发现结果抖动。
- 新复制语义：支持黑名单、受限向量时钟或其他冲突元数据的语义断言。

完成标准：

- 后续能力默认遵循“先补夹具和断言，再补实现，再纳入固定回归集”的顺序。

## 覆盖清单与验收标准

### 覆盖清单

增强后的测试体系至少需要稳定覆盖以下场景：

- 双节点复制、补拉、重复投递幂等。
- 晚加入节点追平、跨多批次补拉、快照修复收敛。
- 相同与不同 `message_window_size` 下的窗口行为。
- 三节点并发写入的字段级收敛与删除优先。
- 网络分区与恢复、乱序到达、广播与补拉并发。
- 时钟超限、未来时间戳拒绝、首次校时前写闸门。
- 多跳路由与登录用户查询。

### 可判定验收标准

- 新增协议特性时，必须先能落入既有三层测试之一，而不是临时拼装专用测试路径。
- 多节点场景不再直接在业务测试中手写监听器、启动流程和基础轮询逻辑。
- 固定回归集覆盖的是复制语义承诺，而不是单条代码路径。
- 测试夹具的抽象不直接依赖 WebSocket 帧格式，未来传输层调整时应能局部适配。

## 风险与非目标

### 当前风险

- 当前仓库测试基线并非完全稳定，已经存在失败用例；因此第一优先级是提升可测试性与回归稳定性，而不是一边重构框架、一边同时修改业务语义。
- 如果在 Phase 1 就强推全量迁移，容易把“夹具重组”做成“测试逻辑重写”，增加回归噪声。
- 如果故障注入过早绑定当前 WebSocket 实现细节，后续做传输抽象时仍会重复返工。

### 非目标

- 不在本轮引入外部测试编排平台或独立集群仿真服务。
- 不把测试框架抽成面向外部复用的公共库。
- 不在没有对应语义文档的情况下，为未来能力预先实现复杂断言逻辑。
